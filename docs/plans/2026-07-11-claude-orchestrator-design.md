# Claude Code オーケストレーターポリシー + カスタムエージェント 設計

日付: 2026-07-11
ステータス: 設計承認済み(実装プラン未作成)

## 目的

Claude Code のメインセッション(Opus / Fable 起動時)をオーケストレーターとして振る舞わせる。
調査・実装・検証はモデルを固定したサブエージェントに委譲し、メインセッションは
ブリーフ作成・結果の評価・統合・報告に徹する。

狙い:

- **コンテキスト衛生**: ファイル探索やビルドログの大量出力をサブエージェント側に隔離し、
  メインセッションの compaction を遅らせて長時間セッションでの判断品質を保つ
- **コスト最適化**: 探索・実行系の作業を haiku / sonnet に落とし、Opus/Fable トークンを
  評価・判断・統合に集中させる
- **並列性**: 独立した調査・作業を同時に走らせられる

## 背景・制約

- メインセッションのモデルは `home-manager/programs/claude-code/default.nix` で
  `model = "opus"` に固定されている。「Opus/Fable のときだけ適用」は実運用上ほぼ常時適用と等価
- Claude Code ハーネスのシステムプロンプトは「ユーザーが頼まない限りエージェントを
  spawn するな」と抑制している。CLAUDE.md のユーザー指示はシステムプロンプトより
  優先されるため、恒常的な委譲許可を明記することで上書きする
- モデル条件分岐は、システムプロンプト内の自己申告("You are powered by ...")を
  プロンプトで参照させることで実現する。hook でのモデル判定は入力仕様がバージョン
  依存のため採用しない

## コンポーネント1: オーケストレーションポリシー

`home-manager/programs/claude-code/user-memory.md` に「Orchestration Policy」
セクションを追加する。内容:

### 適用条件

- 自分のモデルが **Opus または Fable** の場合のみ適用(システムプロンプトの自己申告で判定)
- Sonnet 以下で起動されている場合は通常通り自分で作業する

### 恒常的な委譲許可

「このポリシーに従ったサブエージェント起動はユーザーの恒常的な指示である」と明記し、
ハーネスの spawn 抑制を上書きする。

### 委譲基準

| 作業内容 | 委譲先 |
|---|---|
| コードベース探索・広域検索・ログ解析 | `explorer` |
| 複数ファイルにまたがる実装・テスト作成 | `worker` |
| ビルド・テスト・動作検証の実行 | `verifier` |

### 委譲しない条件(過剰委譲の防止)

- 単発の質問への回答
- 1〜2 ファイルの小さな変更
- ユーザーとの要件すり合わせ
- 直前の文脈に強く依存する対話的デバッグ

### メインセッションの責務

- 自己完結したブリーフの作成(サブエージェントはこの会話を見られない前提で、
  ファイルパス・受け入れ条件・制約を含めて書く)
- 返ってきた結果の批判的検証(鵜呑みにしない)
- 結果の統合とユーザーへの報告
- 結果が不十分な場合は `SendMessage` で同一エージェントに追加指示する(再 spawn しない)

### 既存資産との棲み分け

- **3 つ以上の独立タスクの並列実行** → `team-task` スキル(既存)
- **実装後のコード改善** → `code-simplifier` / `code-review`(user-memory.md の既存ルールを維持)
- **それ以外の日常タスク** → 本ポリシー

## コンポーネント2: カスタムエージェント定義

`home-manager/programs/claude-code/agents/` を新設し、`default.nix` のコメントアウト
されている `agentsDir = ./agents;` を有効化する。

| ファイル | model | tools | 役割 |
|---|---|---|---|
| `explorer.md` | haiku | Read, Glob, Grep, Bash(読み取り想定) | 調査専用。結論と参照パスだけ返す |
| `worker.md` | sonnet | 制限なし | 実装・修正・テスト作成 |
| `verifier.md` | sonnet | Bash, Read 中心 | ビルド・テスト実行と結果の事実報告 |

各エージェントの system prompt に共通で含める指示:

- 報告は要点のみ。ファイルダンプや長大なログをそのまま返さない
- 証拠(実行したコマンドと出力の要旨、参照したファイルパス)を添える

## データフロー

```
ユーザー依頼
  → メイン (Opus/Fable) が委譲基準で判定
      ├─ 基準に該当 → ブリーフ作成 → Agent tool で委譲
      │     → サブエージェントの最終メッセージをメインが検証・統合 → 報告
      └─ 閾値未満 → メインが直接作業
```

## 検証

1. `nix fmt`
2. 新規ファイルを `git add`(Nix Flakes は git 追跡ファイルしか見えない)
3. 両ホストのビルド確認:
   - `nix build .#darwinConfigurations.kohei-m4-mac-mini.system --no-link`
   - `nix build .#darwinConfigurations.SC-N-843.system --no-link`
4. `darwin-rebuild switch` 適用後、`~/.claude/agents/` への symlink を確認
5. 新セッションで Agent tool の利用可能タイプに explorer / worker / verifier が
   載ることを確認し、実際に委譲が起きるか観察する

## やらないこと

- hook による強制・毎ターン注入(案C)— 本構成の運用で遵守率に不満が出てから検討
- opus 固定の reviewer エージェント — メイン自身が評価者なので冗長
- 既存の team-task / code-simplifier ルールの変更 — 棲み分けの明文化のみ

## 検討済みの代替案

- **案A: プロンプトポリシーのみ** — 実装コスト最小だが、委譲先のモデル指定ができず
  要件の中核を満たせない
- **案C: 案B + UserPromptSubmit hook 注入** — 遵守率は最大だが、毎ターンのノイズと
  トークン消費が増える。CLAUDE.md が毎セッション読み込まれることを考えると重複投資
