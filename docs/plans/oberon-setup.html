Oberon セットアップ手順

さくらのVPS 上に NixOS + Forgejo を構築し、Cloudflare Tunnel 経由で forgejo.thinceller.dev と oberon.thinceller.dev を 自分だけアクセスできる形 (Cloudflare Access) で公開するための、 リポジトリ scaffolding 完了後の作業手順書です。

前提 dotfiles リポジトリへの初期コミット (nixosConfigurations.oberon、 nixos/modules/*.nix、hosts/oberon/*.nix、Mac 側 cloudflared / SSH config 追加) は完了済みです。
このドキュメントでは 残っている手動作業 と、その合間に行う プレースホルダの置換 を、上から順に実行できる粒度で説明します。

目次

  1. 必要な情報・準備物の確認
  2. Mac の SSH 公開鍵を common.nix に貼り付け
  3. Cloudflare Tunnel を作成
  4. Cloudflare Access で Forgejo Web UI と SSH を保護
  5. トンネル UUID を cloudflared.nix に書き込む
  6. nixos-anywhere で NixOS を投入
  7. VPS の age 公開鍵を取得して sops を有効化
  8. Credentials を暗号化し cloudflared.nix の sops 統合を有効化
  9. 最初の nixos-rebuild (cloudflared 起動)
  10. 動作確認 (HTTPS / トンネル SSH)
  11. Forgejo の管理者ユーザ作成
  12. git CLI 用 Service Token を発行し Mac 側 git を設定
  13. ロックダウン (sshd を 127.0.0.1 バインドに切替)
  14. 日常運用コマンド
  15. トラブルシューティング

Step 0必要な情報・準備物の確認

作業を始める前に、以下が手元に揃っていることを確認してください。

項目 内容・取得方法
VPS の グローバル IP アドレス さくらインターネットのコントロールパネル → サーバー詳細
VPS への 初回 SSH 鍵 さくらVPS 契約時に登録した公開鍵に対応する Mac 側の秘密鍵。デフォルトは ~/.ssh/id_ed25519
VPS の 初期ユーザ名 Ubuntu 24.04 の場合は ubuntu または root。ssh ubuntu@<IP> または ssh root@<IP> で疎通を確認
Cloudflare アカウント thinceller.dev のネームサーバが Cloudflare を向いていること
Mac 側の age 秘密鍵 ~/.config/sops/age/keys.txt に既存。これが無いと sops 復号化ができません。 age-keygen -y ~/.config/sops/age/keys.txt で公開鍵が確認できます。

また、dotfiles 側で次の変更が 新しい darwin-rebuild switch で適用済み であることを確認してください。Mac 側に cloudflared コマンドが入っていないと、後段のトンネル経由 SSH ができません。

which cloudflared
# /run/current-system/sw/bin/cloudflared (またはhome-managerのprofile配下)が出ればOK

# まだの場合:
sudo darwin-rebuild switch --flake .#kohei-m4-mac-mini

Step 1Mac の SSH 公開鍵を common.nix に貼り付け

nixos/modules/common.nix の openssh.authorizedKeys.keys にはプレースホルダが入っています。これを Mac の実際の公開鍵で置き換えます。

  1. Mac で公開鍵を表示する: 
    cat ~/.ssh/id_ed25519.pub
    例: ssh-ed25519 AAAAC3Nz...XYZ thinceller@kohei-m4-mac-mini
  2. nixos/modules/common.nix を開き、以下の行を置き換える: 
    - "ssh-ed25519 AAAA_REPLACE_WITH_MAC_PUBLIC_KEY thinceller@mac"
+ "ssh-ed25519 AAAAC3Nz...XYZ thinceller@kohei-m4-mac-mini"
補足 公開鍵は 公開して問題ない情報 なので、そのまま git にコミットしても構いません。 逆に、ここを置換し忘れると nixos-anywhere による初回投入の後、自分自身が SSH で入れなくなるので必ず置換してください。

Step 2Cloudflare Tunnel を作成

  1. dash.cloudflare.com にログインし、左メニューの Zero Trust をクリックします。 (初回は Zero Trust の team 名を聞かれます。任意の名前で OK。無料プランで足ります。)
  2. Networks → Tunnels → Create a tunnel
  3. コネクタは Cloudflared を選択 → Next
  4. Tunnel name に oberon と入力し、Save tunnel。
  5. 次の画面で OS 別のインストールコマンドが表示されますが、 そのままスクロールして無視 してください。 本構成では cloudflared を NixOS 上で services.cloudflared が管理するため、ここでは何も実行しません。
    ただし、画面に表示されるトークンを含む URL を控えておく必要はありません。代わりに次の手順で Credentials JSON を取得します。 Next をクリックして進む。
  6. Public Hostnames 画面で次の 2 行を登録します:
    SubdomainDomainTypeURL
    gitthinceller.devHTTPlocalhost:3000
    oberonthinceller.devSSHlocalhost:22
    2 つ目を追加するには + Add a public hostname をクリック。
  7. Save tunnel をクリック。これでトンネルが作成されます。
  8. Credentials JSON とトンネル UUID を取得する:
    作成完了後の画面右上の ... → Configure でトンネル詳細に入り、左メニュー Overview でトンネル ID (UUID) を確認、メモします。
    Credentials JSON はトンネル作成画面では直接ダウンロードできないため、次の手順で取得します。

Credentials JSON を取得する別経路

Cloudflare の Web UI からは Credentials JSON ファイルを直接ダウンロードできないため、 Mac 上で cloudflared CLI を使ってトンネルにログイン → JSON を取得します。

  1. cloudflared を Cloudflare アカウントと紐付ける: 
    cloudflared login
    ブラウザが開き、thinceller.dev のゾーンを選択して認可します。完了後 ~/.cloudflared/cert.pem が作成されます。
  2. 既存トンネル oberon の Credentials を吐き出す: 
    cloudflared tunnel token --cred-file /tmp/cf-tunnel.json oberon
    /tmp/cf-tunnel.json に {"AccountTag", "TunnelID", "TunnelSecret"} 形式の JSON が書き出されます。 この内容は後で sops で暗号化するので、終わったら必ず削除します。
補足 上記 cloudflared tunnel token が UUID を含む JSON 出力にならない場合、 cloudflared tunnel list で UUID を確認し、 cloudflared tunnel info <UUID> でも詳細が見られます。

Step 3Cloudflare Access で Forgejo Web UI と SSH を保護

oberon.thinceller.dev (SSH) と forgejo.thinceller.dev (Forgejo Web UI / git HTTPS) の 両方 を、自分のメールアドレスでのみ認可される Self-hosted アプリで保護します。 git CLI からの clone/push にはブラウザ認証が使えないため、 Service Token (Client ID / Secret) も発行して、Forgejo Web アプリのポリシーに追加します。

3-1. Oberon SSH アプリを作成

  1. Zero Trust → Access → Applications → Add an application
  2. Self-hosted を選択
  3. Configuration:
    • Application name: Oberon SSH
    • Session Duration: 24 hours (好みで)
    • Application domain: oberon.thinceller.dev
  4. Identity providers: One-time PIN (デフォルト) のままで OK
  5. Add policies →
    • Policy name: allow-self
    • Action: Allow
    • Configure rules → Emails → is → thinceller@gmail.com
  6. 残りの画面はデフォルトのまま Next → Add application。

3-2. git CLI 用 Service Token を発行

git CLI は OTP メール認証フローを通れないため、HTTP ヘッダで認証を渡す Service Token を Forgejo Web アプリに紐付けます。先にトークンだけ作っておきます。

  1. Zero Trust → Access → Service Auth → Service Tokens → Create Service Token
  2. Configuration:
    • Service Token Name: forgejo-cli
    • Service Token Duration: 1 year (好みで、最長は Cloudflare 側に従う)
  3. 表示される Client ID と Client Secret を安全な場所に控える。 Secret はこの画面でしか表示されないため、必ずコピーしてください。 後で 1Password などのパスワードマネージャに保管します。

3-3. Forgejo Web アプリを作成

  1. Zero Trust → Access → Applications → Add an application
  2. Self-hosted を選択
  3. Configuration:
    • Application name: Forgejo Web
    • Session Duration: 24 hours
    • Application domain: forgejo.thinceller.dev
  4. Identity providers: One-time PIN (デフォルト) のままで OK
  5. Add policies → 2 つのポリシーを別々に作成:
    • 1 つ目 (ブラウザ用):
      • Policy name: allow-self-email
      • Action: Allow
      • Configure rules → Emails → is → thinceller@gmail.com
    • 2 つ目 (git CLI 用):
      • Policy name: allow-service-token
      • Action: Service Auth (Allow ではなく Service Auth を選ぶことで、メール認証フローをスキップし Service Token のみで通すポリシーになります)
      • Configure rules → Service Token → is → forgejo-cli
  6. Next → Add application。
補足 Cloudflare Access のポリシーには 2 種類あります:
  • Allow: 通常の人間ユーザ向け。Identity Provider (メール OTP 等) での認証が必要。
  • Service Auth: 自動化用。CF-Access-Client-Id / CF-Access-Client-Secret HTTP ヘッダだけで通る。
Forgejo Web アプリは両方持たせることで、「ブラウザでは自分のメールで、CLI では Service Token で」というデュアル運用ができます。

Step 4トンネル UUID を cloudflared.nix に書き込む

nixos/modules/cloudflared.nix を開き、UUID プレースホルダを実際の値に置換します。

- "REPLACE_WITH_TUNNEL_UUID" = {
+ "12345678-aaaa-bbbb-cccc-1234567890ab" = {

この段階ではまだ sops.secrets."cloudflared" ブロックはコメントアウトされたままで OK です (sops 統合は Step 7 で有効化)。

Step 5nixos-anywhere で NixOS を投入

さくらVPS の Ubuntu を NixOS で上書きします。nixos-anywhere は SSH 経由で VPS に kexec NixOS インストーラを送り込み、disko でディスクをパーティショニングし、NixOS を書き込みます。

破壊的操作 この手順は VPS のディスクを disko によって完全にワイプ・再パーティショニング します (hosts/oberon/disko.nix の宣言通り、GPT + 1MiB BIOS boot partition + ext4 root)。 実行前に、VPS 上に残したいデータが無いことを確認してください。
さくらのVPS は静的 IP 構成 (DHCP 不在) さくらのVPS は DHCP を提供せず、コンパネで割り当てられた静的 IP / netmask / gateway / DNS を OS 側で設定する運用。Ubuntu/CentOS では cloud-init が自動投入しますが、 NixOS には cloud-init が無いので hosts/oberon/network.nix で値を宣言する必要があります。
このファイルは .gitignore 済みです — origin IP を public repo に晒すと cloudflared で隠蔽している意味がなくなるため。 テンプレートは hosts/oberon/network.nix.example。
コンパネ → 該当VPS → "ネットワーク" タブに表示される アドレス / ネットマスク / ゲートウェイ / DNS を network.nix に 転記してください。これを忘れると install 完了・boot しても NIC に IP が割り当たらず、外から到達不能になります。
Flakes は untracked ファイルを見ない network.nix は .gitignore されているので、そのままでは nix flake 系コマンドや nixos-anywhere から見えません。 毎回 nixos-anywhere / nixos-rebuild の前に: 
git add -fN hosts/oberon/network.nix
で intent-to-add (force) しておくと、flake は見るが git commit 対象にはならない状態になります。
ただし git commit -a (-a で全変更を自動 stage する形) を 使うとこのファイルも巻き込まれる ので、このリポジトリでは -a は使わず明示的に git add <path> で stage する運用にしてください。
さくらのVPS は SeaBIOS (legacy BIOS) なので注意 さくらのVPS の firmware は UEFI ではなく SeaBIOS です。 bootloader は systemd-boot ではなく GRUB を BIOS モードで使う必要があります (hosts/oberon/configuration.nix の boot.loader.grub.enable = true)。
disko 側も ESP (vfat, EF00) ではなく BIOS boot partition (1MiB, EF02) を GPT の最初に置くこと。
これらが噛み合っていないと、install 自体は完走するが reboot 後に SeaBIOS の起動エントリで ハングして VPS が起動しません (Booting from Hard Disk... から先に進まない)。
なぜ disko と --build-on remote が必要か 現行版の nixos-anywhere は disko によるディスク宣言を必須 とします (config.system.build.diskoScript を要求)。
また Mac (aarch64-darwin) から x86_64-linux のクロージャを作るには Linux ビルダーが必要ですが、 未整備の状態では --build-on remote を付けて ビルドを VPS 側で実行 させるのが最短ルートです。Mac 側はあくまで eval だけを行います。
重要: bootstrap モードで初回 deploy する nixos/modules/common.nix の冒頭に bootstrap フラグがあります。 初回 deploy 時は必ず bootstrap = true のままにすること (デフォルトもtrueです)。
true の間は sshd が 0.0.0.0:22 にバインドされ、firewall も 22 番だけ開きます。 これは Step 6 で age 公開鍵を取得 (ssh-keyscan <VPS_IP>) するために必須です。
false にすると sshd は 127.0.0.1 専用 + firewall 全閉となり、cloudflared 経由でしか到達できなくなります。 これは Step 9 の動作確認後、最後の Step 12 で行うロックダウン手順です。
VPS は NixOS stable channel を使う oberon は flake.nix の nixpkgs-stable (現行 25.11) を参照しています。 nixpkgs-unstable は Hydra のビルドキャッシュが遅れることがあり、 ソースビルドフォールバックされると 2GB RAM の kexec installer (tmpfs) 上で OOM します (典型例: glibc-locales 解凍中の xz が SIGKILL される)。
また i18n.defaultLocale を独自に変更しないこと。 supportedLocales が派生で変わると glibc-locales のキャッシュキーがズレて確実にソースビルドになります。 time.timeZone は無関係なので Asia/Tokyo にして OK。
  1. 変更を git に staging する: 
    cd ~/.dotfiles
git add nixos hosts/oberon flake.nix flake.lock \
        home-manager/programs/ssh \
        home-manager/programs/default.nix \
        home-manager/modules/pkgs/common.nix

# network.nix は gitignore 済みなので intent-to-add (force) で flake に見せる
git add -fN hosts/oberon/network.nix

    Nix Flakes は git に追跡されていないファイルを認識しません。git add だけで OK (commit はまだしなくて良い)。network.nix だけは .gitignore 済みなので -fN (force + intent-to-add) が必須。

  2. nixos-anywhere を実行: 
    nix run github:nix-community/nixos-anywhere -- \
  --flake .#oberon \
  --generate-hardware-config nixos-generate-config \
    ./hosts/oberon/hardware-configuration.nix \
  --build-on remote \
  --target-host ubuntu@<VPS_IP>

    初期OS が root しか SSH 許可していない場合は ubuntu@ を root@ に置き換えてください。
    --generate-hardware-config で VPS 上の nixos-generate-config 出力が hosts/oberon/hardware-configuration.nix に書き戻されます (fileSystems は disko が別途設定するので、ここでは kernel modules / hostPlatform 等だけが入ります)。
    --build-on remote はビルドを VPS 上で行うフラグ (--build-on-remote は deprecated エイリアス)。Mac の linux-builder 未整備でも通ります。

  3. フローは kexec → disko (ディスクワイプ) → install → reboot の順で進みます。 所要時間はおおむね 15〜30 分。Successfully installed NixOS の表示で完了。
  4. 生成された hardware-configuration.nix を commit: 
    git add hosts/oberon/hardware-configuration.nix
git diff --cached hosts/oberon/hardware-configuration.nix  # 実機の情報に置き換わっていることを確認
再実行する場合 (失敗・やり直し時) ケース1: kexec installer にまだ繋がる (失敗が install 前のフェーズ):
kexec 後は SSH ホスト鍵が変わっているため ssh-keygen -R <VPS_IP> して、 --target-host root@<VPS_IP> に切り替えて再実行すると kexec フェーズを スキップして disko 以降から再開できます。

ケース2: install 完走後にやり直したい / 設定ミスでブリック (例: bootstrap=false で初回 deploy してしまい外部 SSH 不可になった):
cloudflared がまだ稼働していない段階では VPS は復旧不能なため、 さくらVPSコンパネ → OS インストール (Ubuntu 24.04) で クリーンに戻し、契約時の SSH 公開鍵を再登録して、ここからやり直すのが最短です。 所要 5〜10 分 + nixos-anywhere ~30 分。

Step 6VPS の age 公開鍵を取得して sops を有効化

NixOS インストール直後、VPS は新しい SSH ホスト鍵を持っています。 この鍵から age 公開鍵を生成し、sops の暗号化対象に追加します。

  1. VPS の SSH ホスト鍵から age 公開鍵を取得: 
    ssh-keyscan <VPS_IP> 2>/dev/null \
  | nix run nixpkgs#ssh-to-age
    出力例: age1abc...xyz
  2. .sops.yaml に追加: 
    keys:
  - &admin age1r9r7z5crn6v77ye6psxvf24mdtmxgnng8xuvdryhckpjpmlyhdyqh8j2pc
  - &sc age1hd03r99ut6kj64y9kxnekggxkzyzav985uzxes35v9e5cfjtp4fs6gelkf
  - &mac age16hjw8qjvc0aakun7e4j9kfpzl787vqcvsfaqyr4u488vlyema5lsu95evr
+ # machines
+ - &oberon age1abc...xyz       # ← Step 6-1 で得た値
creation_rules:
  - path_regex: secrets/[^/]+\.(yaml|json|env|ini)$
    key_groups:
      - age:
          - *admin
          - *sc
          - *mac
+         - *oberon
  3. 既存の暗号化済みファイルを新しい鍵セットで再暗号化: 
    sops updatekeys secrets/default.yaml

    これで secrets/default.yaml も oberon の鍵で復号できるようになります (今回 oberon は cloudflared.json しか使いませんが、運用上 .sops.yaml は単一の真実なので統一しておきます)。

  4. VPS 側の sops 鍵設定について:
    sops-nix は sops.age.sshKeyPaths (デフォルト [/etc/ssh/ssh_host_ed25519_key]) から age 鍵を 毎回 on-the-fly に派生 して secret を復号します。永続的な age 鍵ファイル (/var/lib/sops-nix/key.txt) は作成されません。
    sops.age.keyFile を明示設定するとそのパスに実体ファイルが必要になりエラーになるので、 明示設定しない (=デフォルトの sshKeyPaths 経由) のが正解。 nixos/modules/sops.nix は空の状態で OK で、利用側 (cloudflared.nix 等) が sops.secrets."<name>" を宣言するだけで Step 8 の nixos-rebuild 時に 自動で復号 + 配置されます。

Step 7Credentials を暗号化し cloudflared.nix の sops 統合を有効化

  1. Step 2 で取得した Credentials JSON を sops で暗号化 (binary モード): 
    cd ~/.dotfiles
cp /tmp/cf-tunnel.json secrets/cloudflared.json
sops encrypt --input-type binary --output-type binary -i secrets/cloudflared.json
rm /tmp/cf-tunnel.json

    binary 形式で暗号化する点が重要。sops --encrypt をデフォルト (json 形式) で実行すると、JSON 内の各値が個別に暗号化されるが、sops-nix 側で sops.secrets."cloudflared" としてアクセスすると「cloudflared という JSON キーを探す」挙動になり the key 'cloudflared' cannot be found エラーになります。
    binary 形式は「ファイル全体を sops エンベロープで包む」だけなので、unwrap 後の /run/secrets/cloudflared には元の JSON がそのまま入り、cloudflared が読めます。

    中身が "sops": {...} を含む暗号化形式になっていることを確認: 
    head -5 secrets/cloudflared.json
sops decrypt --input-type binary --output-type binary secrets/cloudflared.json | head -3
# → 元の JSON {"AccountTag":"...","TunnelID":"...","TunnelSecret":"..."} が返る
  2. nixos/modules/cloudflared.nix を編集:

    冒頭のコメントに従い、次の 2 箇所を変更します。

    • sops.secrets."cloudflared" ブロックのコメントを解除: 
      - # sops.secrets."cloudflared" = {
- #   sopsFile = ../../secrets/cloudflared.json;
- #   format = "json";
- #   owner = config.services.cloudflared.user;
- #   group = config.services.cloudflared.group;
- # };
+ sops.secrets."cloudflared" = {
+   sopsFile = ../../secrets/cloudflared.json;
+   format = "binary";   # JSON ファイル全体を unwrap (特定キー抽出ではない)
+   mode = "0400";        # NixOS 25.11 で services.cloudflared が DynamicUser 化
+                         # → owner/group 指定不可。root:root 0400 で systemd LoadCredential 経由
      ファイル冒頭の関数引数も { config, ... }: に変更してください (現在は { ... }:)。
    • credentialsFile をプレースホルダから sops の path に切り替え: 
      - # credentialsFile = config.sops.secrets."cloudflared".path;
- credentialsFile = "/run/secrets/cloudflared"; # sops 統合後に上記行へ置換
+ credentialsFile = config.sops.secrets."cloudflared".path;
  3. secrets ファイルも staging: 
    git add secrets/cloudflared.json .sops.yaml
git add nixos/modules/cloudflared.nix
git add secrets/default.yaml  # updatekeys で書き換わったため
  4. Mac 側でeval が通ることを確認: 
    nix eval .#nixosConfigurations.oberon.config.system.build.toplevel.drvPath

Step 8最初の nixos-rebuild (cloudflared 起動)

この時点では cloudflared がまだ起動していないので、VPS の IP に直接 SSH してビルド・スイッチを行います。 クロスビルドを避けるため、--build-host も同じ VPS を指します。 

nixos-rebuild switch \
  --flake .#oberon \
  --target-host thinceller@<VPS_IP> \
  --build-host thinceller@<VPS_IP> \
  --use-remote-sudo
  1. SSH 接続で初回フィンガープリント確認 → yes。
  2. VPS 側で nixpkgs を取得し、ビルドが走ります (~5〜15 分)。
  3. switch 完了後、VPS 上で各サービスの起動を確認: 
    ssh thinceller@<VPS_IP> systemctl status forgejo cloudflared postgresql
    3 つすべて active (running) であれば成功。
  4. Cloudflare ダッシュボード → Zero Trust → Networks → Tunnels でトンネル oberon の Status が HEALTHY になっていることを確認。

Step 9動作確認 (HTTPS / トンネル SSH)

  1. HTTPS Web UI 確認:
    ブラウザで https://forgejo.thinceller.dev を開く。 初回は Cloudflare Access の認証画面が出るので、メールアドレスを入力 → 届いた One-time PIN を入力。 通過後に Forgejo の初期セットアップ画面 (もしくは "Install Forgejo" ページ) が表示されれば、 Cloudflare Access → cloudflared → Forgejo の経路が正しく繋がっています。
  2. トンネル経由 SSH を確認: 
    ssh oberon
    matchBlocks."oberon" が ProxyCommand に cloudflared access ssh を仕込んでいるため、初回はブラウザが開いて Cloudflare Access の認証フロー (One-time PIN メール) が走ります。
    メールで届いた 6 桁コードを入力 → 認証完了後、自動で SSH が確立します。
  3. 2 回目以降は無認証で繋がる:
    Access のセッション (デフォルト 24h) が残っているため、しばらくは ssh oberon だけで入れます。
  4. nixos-rebuild もトンネル経由に切り替える: 
    nixos-rebuild switch \
  --flake .#oberon \
  --target-host oberon \
  --build-host oberon \
  --use-remote-sudo
    以降の運用では IP は触りません。
ここまでで Web/SSH の Access 認証経路は完成 「インバウンドポートゼロ + Cloudflare Access で自分のメールアドレスだけが Forgejo Web UI と SSH に到達できる」 設計が稼働しています。VPS の IP・ポート・秘密鍵はリポジトリにも公開ネットワークにも露出していません。
ただし git clone / git push をコマンドラインから行うには Step 11 の Service Token 設定がまだ必要です。

Step 10Forgejo の管理者ユーザ作成

DISABLE_REGISTRATION = true を設定しているため、Web UI からの自己登録はできません。 最初の管理者ユーザは VPS 上で CLI 経由で作成します。

nixos/modules/forgejo.nix で forgejo-admin ラッパー (内部で sudo -u forgejo forgejo --config /var/lib/forgejo/custom/conf/app.ini --work-path /var/lib/forgejo "$@" に展開) を 環境に入れているので、--config や sudo -u を毎回書く必要はありません。 

ssh oberon

# VPS 上で実行
forgejo-admin admin user create \
  --admin \
  --username thinceller \
  --email thinceller@gmail.com \
  --random-password \
  --must-change-password

出力されたランダムパスワードを保存しておき、https://forgejo.thinceller.dev/user/login からログイン (Cloudflare Access 認証通過後) → 強制パスワード変更を行ってください。

他にも forgejo-admin admin user list / forgejo-admin dump / forgejo-admin migrate などの運用コマンドがそのまま使えます。

Step 11git CLI 用 Service Token を発行し Mac 側 git を設定

Forgejo Web アプリを Cloudflare Access で保護しているため、 git clone https://forgejo.thinceller.dev/... や git push はそのままでは 302 で Access のログイン画面にリダイレクトされて失敗します。 Step 3-2 で発行した Service Token の Client ID / Secret を http.https://forgejo.thinceller.dev/.extraHeader で送るよう gitconfig を設定し、 機密は sops で暗号化して dotfiles で管理します。

11-1. include 用 gitconfig ファイルを作成して sops 暗号化

Service Token の Client ID / Secret は 機密なので、 平文を dotfiles に commit しないよう sops 暗号化したファイルとして配置します。 

cd ~/.dotfiles

# (1) 一時的に平文で作成し、Service Token の値を埋める
cat > secrets/cloudflare-access.gitconfig <<'EOF'
[http "https://forgejo.thinceller.dev/"]
  extraHeader = CF-Access-Client-Id: PASTE_CLIENT_ID_HERE
  extraHeader = CF-Access-Client-Secret: PASTE_CLIENT_SECRET_HERE
EOF
$EDITOR secrets/cloudflare-access.gitconfig   # PASTE_..._HERE を実値に置換

# (2) sops binary 形式で暗号化 (ファイル全体を unwrap して home-manager 側で復号する形)
sops encrypt --input-type binary --output-type binary -i secrets/cloudflare-access.gitconfig

# (3) 暗号化されたことを確認
head -5 secrets/cloudflare-access.gitconfig
sops decrypt --input-type binary --output-type binary secrets/cloudflare-access.gitconfig
# → 元の [http "..."] 構造が返れば成功

.sops.yaml には専用の creation_rule があり、この cloudflare-access.gitconfig は oberon (VPS) を recipient に含めない 構成です — VPS 側で git CLI は使わないのでアクセス権を絞っています。

11-2. home-manager で復号 + git の include を有効化

home-manager/programs/git/default.nix に既に組み込まれています: 

sops.secrets."cloudflare-access.gitconfig" = {
  sopsFile = ../../../secrets/cloudflare-access.gitconfig;
  format = "binary";
  path = "${config.home.homeDirectory}/.config/git/cloudflare-access.gitconfig";
};

programs.git.settings.includeIf."hasconfig:remote.*.url:https://forgejo.thinceller.dev/**".path =
  "${config.home.homeDirectory}/.config/git/cloudflare-access.gitconfig";

activation 時に sops-nix が復号して ~/.config/git/cloudflare-access.gitconfig に symlink を作成、git は hasconfig:remote.*.url:https://forgejo.thinceller.dev/** に該当するリポジトリでのみその設定を include します。 

git add .sops.yaml secrets/cloudflare-access.gitconfig home-manager/programs/git/default.nix
sudo darwin-rebuild switch --flake .#kohei-m4-mac-mini

11-3. 動作確認

# symlink が貼られているか
ls -la ~/.config/git/cloudflare-access.gitconfig
# → ~/.config/sops-nix/secrets/cloudflare-access.gitconfig への symlink

# 中身が復号されているか (extraHeader 行が見える)
cat ~/.config/git/cloudflare-access.gitconfig

# 実際に clone (空の Forgejo リポジトリを Web UI で作成しておく)
cd /tmp && git clone https://forgejo.thinceller.dev/thinceller/test.git
cd test
git config --get-all 'http.https://forgejo.thinceller.dev/.extraHeader'
# → CF-Access-Client-Id / CF-Access-Client-Secret の 2 行が返れば include 経由で効いている

403 / "Access denied" の場合は Forgejo Web アプリの Service Auth ポリシーで forgejo-cli Service Token が許可されているか再確認。 ブラウザの Cloudflare 認証画面が代わりに走る場合は extraHeader が送られていない (= sops 復号失敗 or include 条件不一致) ので git config --get-all 'http.https://forgejo.thinceller.dev/.extraHeader' で確認。

Step 12ロックダウン (sshd を 127.0.0.1 バインドに切替)

ここまでで Cloudflare Access 経由の SSH (ssh oberon) と Forgejo Web UI / git CLI が全部動作しているはずです。最後に common.nix の bootstrap トグルを false にして、外部から VPS の 22 番ポートに直接届かない状態 (= 公開ポートゼロ) にロックダウンします。

注意 この切替後は VPS IP に直接 SSH できなくなります。 cloudflared / Cloudflare Access が壊れると緊急アクセスは さくらコンパネ → VNC コンソール のみになります。Step 9 で ssh oberon が問題なく通ることを確認してから実行してください。
  1. nixos/modules/common.nix の bootstrap = true; を bootstrap = false; に変更: 
    - bootstrap = true;
+ bootstrap = false;
  2. cloudflared 越しで再 deploy: 
    git add nixos/modules/common.nix
nixos-rebuild switch \
  --flake .#oberon \
  --target-host oberon \
  --build-host oberon \
  --use-remote-sudo

    --target-host oberon (= oberon.thinceller.dev via cloudflared) を使うので、 ロックダウン適用後も既存セッションは維持されます。

  3. ロックダウン確認: 
    # 外部から直接届かないことを確認
nmap -p 22 <VPS_IP> -Pn          # 22/tcp filtered または closed

# cloudflared 経由は通ることを確認
ssh oberon echo ok               # → ok
curl -I https://forgejo.thinceller.dev   # → 200 OK (Access の認証付き)
完成 (最終状態) インバウンドポートゼロ、Cloudflare Access で自分のメールアドレスのみが Forgejo Web UI と SSH に到達可能、 git CLI は Service Token 経由でアクセス可能。VPS の IP・ポート・秘密鍵はリポジトリにも公開ネットワークにも露出していません。

git clone が 403 / "Access denied" になる

  2. Zero Trust → Access → Applications → Forgejo Web に Service Auth アクションのポリシーがあり、Service Token forgejo-cli が紐付いているか
  3. Service Token の有効期限切れ → 新規発行して extraHeader を更新
